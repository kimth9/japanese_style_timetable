[레일블루(rail.blue) 열차 운행 계획 데이터 크롤링 기술 노하우 정리]

1. 기술적 분석 배경
   - 사이트 특성: 레일블루는 클라이언트 사이드 렌더링(CSR) 방식을 사용하는 개인 운영 사이트임.
   - 초기 접근 실패: 일반적인 requests + BeautifulSoup을 이용한 정적 크롤링 시 데이터가 포함되지 않은 HTML 껍데기만 수집됨.
   - 데이터 원천 확인: 브라우저 개발자 도구(F12) 네트워크 탭 분석을 통해 내부 AJAX API 엔드포인트 발견.

2. 핵심 데이터 엔드포인트 (Internal AJAX API)
   - URL: https://rail.blue/railroad/logis/getscheduleinfo.aspx
   - 필수 파라미터:
     * u: 1 (사용자 구분값으로 추정)
     * company: (공백일 경우 코레일 일반열차 기준)
     * train: 열차번호 (URL 호출 시에는 정수형으로 전달하는 것이 안정적, 예: 00001 -> 1)
     * date: 조회 날짜 (YYYYMMDD 형식)
     * json: 1 (데이터를 JSON 구조로 받기 위한 필수 설정)
     * version: 20180415 (사이트 고유 버전 스탬프)

3. 크롤링 구현 핵심 로직 (Python 기반)
   - 세션 관리: requests.Session()을 사용하여 커넥션을 유지하고 브라우저 환경과 유사하게 헤더 설정.
   - 예시 코드:
     ```python
     import requests
     session = requests.Session()
     url = "https://rail.blue/railroad/logis/getscheduleinfo.aspx"
     params = {'u': '1', 'train': '1', 'date': '20251209', 'json': '1'}
     response = session.get(url, params=params, timeout=60)
     data = response.json()
     ```

4. 데이터 필터링 및 전처리 노하우
   - 정차역 구분: 응답 JSON 내 's' 배열의 각 요소 중 'stop' 필드 분석.
     * 'dept': 출발역 (도착시간 없음)
     * 'stop': 중간 정차역 (도착/출발시간 모두 존재)
     * 'dest': 종착역 (출발시간 없음)
     * 'skip': 통과역 (데이터 수집 시 무시)
   - 인코딩 주의사항: 응답은 UTF-8이나, 일부 환경에서 깨져 보일 수 있음. hex 값 확인을 통해 데이터 정합성 검증 필요.

5. 안티-크롤링 및 차단 회피 전략 (핵심)
   - 타임아웃 최적화: 서버 응답이 느린 경우가 잦으므로 최소 60초 이상의 Timeout 설정 권장.
   - 지능형 대기 (Human-like Behavior):
     * 일반 대기: 요청 간 10~20초 사이의 랜덤 지연시간 부여.
     * 긴 휴식 (Big Break): 50대 조회마다 3~5분간의 대규모 휴식 시간 추가.
   - 지수 백오프(Exponential Backoff): Timeout 발생 시 30초, 60초, 120초 단위로 대기 시간을 늘리며 최대 3회 재시도.
   - 세션 초기화: 통신 오류 발생 시 세션을 새로 생성하여 TCP 연결 문제를 해결.

8. 결론 및 윤리적 고려사항
   - 레일블루는 개인 운영 사이트이므로 과도한 요청은 서버에 치명적인 부담을 줄 수 있음.
   - 위에서 제시한 지능형 대기 로직을 반드시 준수하여 서버 운영자에게 피해를 주지 않는 범위 내에서 데이터를 수집해야 함.
   - 가능하다면 공공데이터포털의 공식 API를 우선 검토하되, 대안이 없을 경우에만 신중하게 크롤링을 수행할 것.
